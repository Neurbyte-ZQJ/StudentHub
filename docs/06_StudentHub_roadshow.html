校园学生工作，散落在 17 张表里。

不是没有系统，而是「同一名学生」，在五个业务条线上被重复录入五次。

// Data Re-entry
68%

  • /01 系统林立。 团员、社团、社区、资助、量化各一套账号、字典、字段，导一份名单要四个 Excel 拼。
  • /02 过程丢失。 推优大会、勤工考勤、量化积分等关键事件靠人工填表，事后无法回放。
  • /03 合规风险。 状态推进缺审计、敏感字段未脱敏、家长开放日拿不出一份「可解释」的过程档案。
  • /04 学生无感。 系统是给老师用的，学生只是被填的对象，主体性几乎为零。
02 · SOLUTION
把「同一名学生」作为唯一索引，
让过程数据自己说话，
让老师回到育人这件事上。

09 · STATEM

状态机，是这套系统的宪法。

任何状态变更都通过 statem.Apply(ctx, from, to, action, payload)，业务代码不允许直接 UPDATE 状态字段。

EVENT FROM ACTION TO GUARD EFFECT
申请受理 S0 submit S1 年龄 14–28 · 资料齐 通知辅导员 · 写 event_log
推优 S1 recommend S2 班级到会 ≥ 2/3 自动归档会议纪要
大会表决 S2 vote S3 实到 / 应到 ≥ 80% 逐人投票结果落库
公示 S3 publicize S4 通过公示稿模板 校内外双渠道发布
备案 S4 archive S5 无异议 · 公示期满 同步到 CMP 量化
10 · SECURITY

把安全与合规做成默认值。

不是上线后再补，而是在「接口契约」与「数据库设计」阶段就被锁死。

// DATA

字段级加密

身份证 / 银行卡 / 家庭经济走 AES-256-GCM，密钥经环境变量注入；列表与导出按角色脱敏。

// BEHAVIOR

全量审计

event_log 追加写、禁物理删除；关键状态变更强制过 audit.Middleware；保留 ≥ 5 年。

// APP

最小可见

细粒度 RBAC，学生只看自己的、辅导员看班级的、院系副书记看全院的；默认拒绝。

// SESSION

令牌与会话

Access 15min + Refresh 7d HttpOnly Cookie；JWT HS256；日志禁打 token / 密码。

// FILE

白名单上传

MIME + 后缀双白名单、单文件 ≤ 50MB、自动重命名；图片走水印。

// DEPLOY

密钥与配置

.env* 不入仓；CI 注入；Dockerfile 多阶段；Nginx 反代静态资源。

CREATE TABLE idx_student (
  id INTEGER PRIMARY KEY AUTOINCREMENT,

  id_card_enc BLOB NOT NULL, -- AES-GCM 密文
